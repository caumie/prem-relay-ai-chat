# 設計指針

- シンプルで明確な実装・必要のない複雑さは避ける・コメントは正しく

# 実装の進め方

- 不明点は確認する。
- 過去の互換性は残さなくてよいです。新形式に移行した際は、関連コードも全部修正する形にします。
- t-wada の TDD を守る。必ず先に近接テスト `*_test.py` を書き、レッド・グリーン・リファクタリングで進める。
- テストには日本語コメントで「観点」と「目的」を短く書く。
- ファイル先頭には責務概要、関数には引数・戻り値・何をなぜ行うかを docstring で書く。
- 型の違和感を `Any` や `cast` で隠さない。設計問題として、必要なら完了済みファイルにも戻って直す。
- 層分離は細かすぎるフォルダ分割より、責務の太いファイル単位を優先する。
- 過去互換のためだけの再エクスポート、旧名ファイル、ヘルパーは残さない。
- route は HTTP/Form/Template/SSE に集中し、認証・チャット操作・LLM接続・永続化は service/repository へ委譲する。

# 問題解決の検討の方法

- この課題に対してどのような現状か
- どのような問題が出ているか
- どうしてそれは解決する必要があるのか
- 解決策の方法の検討
- 解決した状態とはいったいどういう状態を指すか
